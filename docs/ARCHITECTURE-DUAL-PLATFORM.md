# WhimprFlow — Dual-Platform Architecture (Windows + macOS)

> Synthesized 2026-07-17 from a 12-agent all-Opus workflow (7 recon tracks, critic, 3 gap-fills, synthesis): dual-platform stack evaluation, a teardown of Wispr Flow's actual Windows build, Windows hotkey/insertion/shell engineering, and cross-platform ASR + LLM benchmarks by hardware tier. Raw tracks in docs/research/win-*.md and docs/research/xplat-*.md.
>
> **This supersedes the "fully native Swift" decision in SPEC.md.** SPEC.md's UI spec, Fn state machine, cleanup rule table, prompts, and feature matrix are unchanged in intent — only the implementation substrate moves to Tauri v2 + a shared Rust core. Read this alongside SPEC.md; the "Deltas" section below lists exactly what changes.


---

## Stack verdict

DECISION: Tauri v2 (Rust shared core + system webview), following Handy's architecture, with TWO deliberate research-forced forks off the pure-one-codebase ideal. || SCORED COMPARISON (weighted /125, from stack-decision track): Tauri v2 ~108 | Rust core + 2 native shells ~95 | Electron ~89 | Flutter ~63. Tauri wins the two decisive highest-weight axes: (a) ONE codebase / dev-velocity / parity (w3) = 5 because Handy is a feature-identical dual-platform reference (MIT), and (b) in-process ONNX ASR + llama.cpp LLM (w3) = 5 because transcribe-rs + llama-cpp-2 run in-process, GPU-accelerated, on both OSes; plus it is the only production-proven transparent, non-activating, click-through, above-fullscreen pill on BOTH macOS and Windows from one codebase (Handy overlay.rs). Electron is the safe heavy middle and Wispr's own choice, but ~800MB RAM (measured in Wispr's Windows build) vs Tauri ~40-120MB idle. Option-4 two-native-shells wins raw fidelity but loses ~13 points to the 2x-UI-codebase penalty against the ideally-ONE-codebase mandate. Flutter disqualified (no in-process ML, no mature Fn/CGEventTap/WH_KEYBOARD_LL-suppression plugin). || PRIOR FULLY-NATIVE-SWIFT DECISION IS KILLED. SPEC.md ARCHITECTURE 'CONFLICT RESOLVED - stack' chose fully-native Swift (AppKit NSPanel/NSStatusItem + SwiftUI NSHostingView). That has no Windows story; keeping it forces a second full native shell in C#/.NET (Wispr's split, which one-codebase forbids). Replaced by Tauri v2 + Rust core + React/webview UI: the same shared-UI-plus-thin-native-shim pattern Wispr uses (shared renderer + per-OS helper), but a smaller shell and a SHARED Rust helper instead of two hand-written ones. || FORK 1 (ASR) reverses the stack track's 'swap FluidAudio to transcribe-rs'. The gap-fill ASR track is decisive: single-engine ort/Parakeet on the CPU EP does NOT reliably meet the perceived ~300-500ms budget on mid-tier Apple Silicon, and the CoreML EP CANNOT run a Parakeet-TDT int8 graph on the ANE (the LSTM prediction net + int8 QDQ ops are unsupported and fall to CPU; the ~125-250-call Rust-driven greedy decoder loop would be a net regression). Measured: FluidAudio ANE = 0.1935s / ~145.8x RTFx (128.6x median, ~190x README) on M4 Pro; best-case ort-CPU Parakeet = ~30x on M4 Max (~400ms e2e) and only ~10-18x on a base M-series Air/M1 (~700-1000ms+ e2e, misses perceived and risks committed <900ms). Handy itself ships Parakeet ort CPU-only on every OS (Cargo.toml has features=['onnx'] only, never ort-coreml; Metal is added ONLY to transcribe-cpp/whisper.cpp). RESOLUTION: KEEP FluidAudio as a Mac-only ASR backend behind a shared AsrEngine trait from v1 (ANE latency+power), with transcribe-rs/ort CPU Parakeet as the Mac portable fallback AND the Windows primary. This forks 'one codebase' only at the ASR-engine seam - the minimal, natural place (Handy already models ASR as a swappable LoadedEngine enum). || FORK 2 (native hooks) chooses Wispr's sidecar over Handy's in-process model for the fidelity-critical native subsystems. The gap-fill native-hook track is decisive: co-locating the WH_KEYBOARD_LL / CGEventTap hook thread in the same process as saturating ONNX+llama.cpp inference creates two risks a dedicated in-process thread does NOT remove: (1) scheduler starvation past the 1000ms LowLevelHooksTimeout / CGEvent tap-timeout window under full-core inference, and (2) an inference crash/OOM kills the hook with the whole process. On Windows the resulting LL-hook removal is SILENT and UNDETECTABLE by design (Microsoft: 'no way for the application to know whether the hook is removed') and unrecoverable; handy-keys only re-hooks on suspend/session events, not on an inference stall. RESOLUTION: run the hook + text-injection + secure-input + AX/UIA context read in a SEPARATE native sidecar process, implemented as ONE shared Rust binary reusing handy-keys 0.3.0 + enigo 0.6.1 on both OSes (isolation with one language, closer to one-codebase than Wispr's Swift+C# split). Windows sidecar = hard requirement; macOS sidecar = strongly advised (in-process is survivable there because the CGEvent tap-timeout is recoverable/self-announcing, so an in-process handy-keys tap with the SPEC.md health watchdog is an acceptable macOS-only fallback if scope must shrink). Shell heartbeats the sidecar and respawns it on hang. || WHAT HAPPENS TO EACH REUSABLE MAC RECIPE: -- NSPanel flags (SPEC.md A): SURVIVE via tauri-nspanel PanelBuilder (ahkohd fork, branch v2.1). PanelLevel::Status = raw NSWindow level 25 (above menu bar); StyleMask borderless().nonactivating_panel(); collectionBehavior can_join_all_spaces().full_screen_auxiliary().ignores_cycle(); no_activate(true), focusable(false), transparent(true), shadow off, decorations(false). Byte-for-byte the OpenSuperWhisper IndicatorWindowManager recipe, reached through a plugin instead of Swift. Windows twin = WebviewWindowBuilder focusable(false) => WS_EX_NOACTIVATE + always_on_top => WS_EX_TOPMOST + force_overlay_topmost() SetWindowPos(HWND_TOPMOST, SWP_NOMOVE|SWP_NOSIZE|SWP_NOACTIVATE|SWP_SHOWWINDOW). -- CGEventTap engine (SPEC.md MACOS IMPLEMENTATION): SURVIVES but MOVES into the shared Rust sidecar. handy-keys uses a direct CGEventTap via objc2_core_graphics (NOT rdev) on a dedicated CFRunLoop thread, consuming (HotkeyManager::new_with_blocking), with a 100ms tap_is_enabled/tap_enable health poll and Fn keycode 0x3F flagsChanged handling. To reach full SPEC fidelity (bare-Fn suppression that kills the OS Globe action, 5s health timer with full remove->recreate->re-add reinstall, and a dedicated stale-held-key watchdog a la Wispr CheckStaleKeys) extend handy-keys or hand-roll a core-graphics/objc2 tap. -- FluidAudio (SPEC.md ASR ENGINE): RETAINED as the Mac primary ASR backend behind AsrEngine (see FORK 1); Apache-2.0 SDK, ANE-only, macOS 14+/iOS 17+, bundles Silero VAD v6.2.1. Called from Rust via a small Swift FFI shim or Swift sidecar; ort CPU Parakeet is the fallback if the CoreML model compile fails. -- llama.cpp embed (SPEC.md CLEANUP LLM RUNTIME): SURVIVES, now as llama-cpp-2 0.1.151 (utilityai/llama-cpp-rs, MIT/Apache-2.0) in the Rust core instead of linking libllama from Swift; features metal (macOS) + vulkan (Windows) both confirmed; loads GGUF, --mlock resident, in-process, no daemon. The optional MLX 'faster decode' toggle from SPEC.md is DROPPED for v1 (Swift-only, marginal for ~67-token outputs). Claude Haiku toggle = a stack-neutral reqwest HTTPS call (x-api-key + anthropic-version: 2023-06-01), exactly Handy's llm_client.rs pattern. || ELECTRON FALLBACK TRIGGER (documented, must be settled on real Win11 hardware): flip to Electron ONLY if the semi-transparent pill renders a black/opaque box AT REST (no resize, no focus change) in the shadow:false + decorations:false + creation-time-transparent + unparented config on any common GPU (Intel iGPU / NVIDIA / AMD) - i.e. Tauri issue #14823-class black edges reproduce on a static 256x46 window. Nothing in the issue corpus shows this without resize/shadow/decorations, but it is unverified for a static pill. Secondary trigger: product requires Wispr's permanently-large transparent hover container AND the grow-on-hover alternative is rejected (then Electron setIgnoreMouseEvents(true,{forward:true}) is marginally cleaner). Neither has been observed to fire.

## Shared core design

PRINCIPLE: maximize the shared Rust core; fork only at three seams the research forces (ASR-engine backend, native-hook/injection sidecar, and per-OS overlay pixel geometry). Handy already forks ~6 platform modules, proving 'one codebase' != 'zero per-OS code'. || SHARED CORE (identical Rust on both OSes, in the Tauri main process): -- State machine: the full IDLE -> RECORDING (push-to-talk) -> FINALIZE -> PASTE -> IDLE plus DOUBLE-TAP -> LOCKED (hands-free) plus Esc -> CANCEL logic from SPEC.md 'FN STATE MACHINE'. Platform-agnostic; fed by trigger tokens from the sidecar. Includes the 20-min session cap, 19-min warning, anti-bounce 500ms cooldown, and single-tap no-op gate. -- Audio pipeline: cpal capture (WASAPI on Windows / CoreAudio on macOS) -> downmix mono -> rubato 48k->16k resample -> vad-rs Silero VAD v6.2.1 (512-sample/32ms frames, threshold 0.5, speech_pad 150ms) -> 16kHz mono f32 buffer. Plus a device-hotplug watcher (cpal does NOT emit these): Windows IMMNotificationClient OnDefaultDeviceChanged / macOS CoreAudio kAudioHardwarePropertyDefaultInputDevice listener, else a re-enumerate poll. -- ASR orchestration: the AsrEngine trait + batch-finalize-on-release logic + long-hands-free rolling 20-30s chunk concatenation. (Backends themselves are per-platform - see seam 1.) -- Cleanup LLM: llama-cpp-2 0.1.151 in-process (Qwen3-4B-Instruct-2507 Q4_K_M, --mlock resident, prefill-on-hotkey-down, streaming) + the CleanupProvider trait with ClaudeCleanupProvider (reqwest, claude-haiku-4-5, thinking OFF, x-api-key). Both providers share the identical system prompt. -- Prompts: the entire WHIMPRFLOW CLEANUP PROMPTS section (system prompt, Level modifiers None/Light/Medium/High, few-shot, verifier, command/rewrite prompt) is data, 100% shared. -- Dictionary: manual vocab + 1 replacement rule/word; Double-Metaphone pre-filter injecting <=15 entries; auto-learn diff engine + wordfreq Zipf>=3.0 gate. Shared logic; only the 'read text around caret / focused element' feed is per-OS (AX on macOS, UIA on Windows - see seam 2). -- Cleanup gates + verifier + raw-ASR fallback: shared. -- Storage: SQLite (transcripts/dictionary/snippets, optionally encrypted) + tauri-plugin-store (settings JSON, replacing UserDefaults). Shared. -- UI: ONE React/webview bundle for all Flow Bar states (idle/listening/recording/locked/transcribing/formatting/processing/done/error), the Hub, Settings, Onboarding. 60fps 5-7-bar waveform computed with rustfft 6.4.0 in Rust, streamed to an HTML canvas throttled to ~30fps (Handy EMIT_THROTTLE_MS=33). This mirrors Wispr's own 'one shared JS bundle with process.platform branches' (113 process.platform refs, 84 win32, 59 darwin in Wispr's main). || SEAM 1 - ASR ENGINE (per-platform backend behind a shared trait): trait AsrEngine { fn transcribe(&self, pcm16k: &[f32]) -> Result<String>; fn warmup(); }. macOS impl = FluidAudioEngine (Swift/CoreML/ANE FFI shim, Parakeet TDT 0.6B v2) with an OrtParakeetEngine CPU fallback. Windows impl = OrtParakeetEngine (transcribe-rs 0.3.8 features=['onnx'], ort =2.0.0-rc.12, Parakeet V2 473MB int8 CPU default; DirectML/CUDA opt-in) with a transcribe-cpp whisper.cpp tiny/base fallback for the AVX2-floor tier. || SEAM 2 - NATIVE SIDECAR (one shared Rust binary, separate process): owns the global hotkey hook (handy-keys 0.3.0 + rdev rustdesk fork), text injection (enigo 0.6.1 + hand-rolled clipboard save/restore), secure-input/elevation detection, and the AX (macOS AXUIElement) / UIA (Windows IUIAutomation) context read + caret-rect read. Length-prefixed JSON over stdio (Wispr's proven contract; codegen the schema like Wispr's quicktype). Handlers mirror Wispr: DictationStart/Stop, PasteText, CheckStaleKeys, UpdateShortcuts, SecureInputStatus, plus Windows-only WindowsKeyUpSimulation and ShiftInsert. macOS-only fallback: the sidecar MAY collapse into the main process (in-process handy-keys tap) if v1 scope demands - Windows may not. || SEAM 3 - OVERLAY GEOMETRY (per-OS pixel tuning, shared React content): platform offset constants differ (Handy: macOS OVERLAY_TOP_OFFSET=46/BOTTOM=15; Windows TOP=4/BOTTOM=40), plus Windows fractional DPI (125/150/175%) and work-area/taskbar math. || CRATE / MODULE LAYOUT: workspace root -> crate whimpr-core (state machine, audio pipeline, ASR orchestration trait, cleanup LLM + providers, prompts, dictionary, gates/verifier, storage) | crate whimpr-asr-ort (transcribe-rs/ort Parakeet, both OSes) | crate whimpr-asr-fluidaudio (cfg(macos) Swift FFI shim) | crate whimpr-llm (llama-cpp-2 wrapper + reqwest Claude client) | crate whimpr-sidecar (bin: handy-keys + enigo + objc2/core-graphics [macos] + windows 0.61.3 [win] + stdio IPC) | crate whimpr-overlay (tauri-nspanel [macos] vs WebviewWindowBuilder+SetWindowPos [win], 3 cfg branches like Handy) | crate whimpr-tauri (Tauri v2 shell, tray, autostart, updater, capabilities) | ui/ (React/TS: overlay, hub, settings, onboarding). Tauri plugins: tauri-plugin-global-shortcut 2.3.1 (fallback only), tauri-plugin-clipboard-manager 2.3.2, tauri-plugin-updater 2.10.0, tauri-plugin-autostart, tauri-plugin-store, tauri-plugin-opener. cfg-gated deps: [target macos] tao patch pin if overlay needs it, objc2, core-graphics; [target windows] windows 0.61.3 (Win32_Media_Audio, Win32_UI_WindowsAndMessaging, Win32_UI_Accessibility), winreg 0.55.

## Windows spec (mirrors the Mac SPEC)

Complete Windows-side spec mirroring the Mac SPEC.md. Target Win10 1809+ / Win11 x64 (and note Win-on-ARM caveats). || HOTKEY ENGINE + DEFAULT COMBO: Default push-to-talk = Ctrl+Win (L/R agnostic), matching Wispr (docs.wisprflow.ai). Hands-free toggle = Ctrl+Win+Space (or double-tap Ctrl+Win). Command Mode = Ctrl+Win+Alt. Cancel = Esc while recording. Do NOT default to Fn on Windows (firmware-invisible on most PCs; a WH_KEYBOARD_LL hook, RawInput, RegisterHotKey, and rdev all fail to see bare Fn except on some ThinkPads via BIOS/WMI) - this is the key Mac(Fn)/Windows(Ctrl+Win) divergence. Engine = a hand-rolled WH_KEYBOARD_LL (idHook=13) hook via SetWindowsHookExW in the shared Rust sidecar (windows 0.61.3 crate), NOT RegisterHotKey (which reports PRESS only, needs a real vk, and cannot express a modifier-only combo) and NOT tauri-plugin-global-shortcut/global-hotkey v0.8.0 (RegisterHotKey-based, 50ms polled synthetic release, no modifier-only, no suppression). Callback on a dedicated hook thread with its own GetMessageW loop; on HC_ACTION read vkCode/flags from KBDLLHOOKSTRUCT, match the tiny trigger set (VK_LWIN 0x5B, VK_RWIN 0x5C, VK_LCONTROL 0xA2, VK_RCONTROL 0xA3, etc.), update an atomic state machine, push a token to a lock-free channel, then IMMEDIATELY return 1 (suppress owned keys) or CallNextHookEx - target single-digit-microsecond callback latency. Handle WM_KEYDOWN 0x0100 / WM_KEYUP 0x0101 / WM_SYSKEYDOWN 0x0104 / WM_SYSKEYUP 0x0105. Start-menu suppression: because Ctrl already dirties the Win press the Start menu usually will NOT open; robustly, once the trigger fires swallow the subsequent Win key-UP (WindowsKeyUpSimulation, exactly Wispr's IPC message). Tag self-injected events with a magic dwExtraInfo and ignore LLKHF_INJECTED (0x10) / LLKHF_LOWER_IL_INJECTED (0x02) to prevent feedback loops. Ship first-class alternatives: Ctrl+Alt, Right Ctrl, a spare key (Pause/ScrollLock/Insert/PageUp), and mouse Mouse4/Mouse5/middle. Hotkey validity rules identical to Mac SPEC (max 3 keys, >=1 modifier or valid mouse button, no L/R mix, no CapsLock, Esc standalone, ~40 reserved combos blocked). Watch session lock/unlock (WTSRegisterSessionNotification, WM_WTSSESSION_CHANGE, WTS_SESSION_LOCK/UNLOCK) to force Idle and re-arm the hook. || INSERTION LADDER (mirrors Mac AX-insert/clipboard ladder): 0. Resolve target at record-start AND re-resolve at delivery: GetForegroundWindow -> GetWindowThreadProcessId -> GetGUIThreadInfo(tid).hwndFocus + IUIAutomation::GetFocusedElement. 1. Guard rung: decline password fields (UIA_IsPasswordPropertyId 30019 == TRUE, or Win32 ES_PASSWORD), and decline+toast elevated targets (target integrity level > ours via OpenProcess+GetTokenInformation(TokenIntegrityLevel); SECURITY_MANDATORY_HIGH_RID 0x3000 vs MEDIUM 0x2000) - UIPI silently eats injection with no error. 2. Rung A - UIA ValuePattern.SetValue (clean, no clipboard) ONLY for known-good native single-line value controls (FrameworkId Win32/WinForm/WPF/XAML), because SetValue REPLACES the whole field and fires no per-keystroke event; verify by re-reading CurrentValue; NEVER for FrameworkId 'Chrome'/Electron (S_OK but no JS onChange) or terminals. 3. Rung B - Clipboard paste (PRIMARY, universal): snapshot prior clipboard (GetClipboardSequenceNumber + all formats) -> SetClipboardData CF_UNICODETEXT (13) + the privacy-exclusion formats ExcludeClipboardContentFromMonitorProcessing / CanIncludeInClipboardHistory=0 / CanUploadToCloudClipboard=0 (registered via RegisterClipboardFormat; keeps dictation out of Win+V history/Cloud Clipboard - the analog of macOS org.nspasteboard.ConcealedType) -> ~30-80ms settle -> SendInput Ctrl+V (VK_CONTROL 0x11 + 'V' 0x56), with Ctrl+Shift+V and Shift+Insert (VK_INSERT 0x2D) as ordered terminal fallbacks -> restore prior clipboard AFTER the paste is consumed via AddClipboardFormatListener/WM_CLIPBOARDUPDATE or a dynamic seqnum check (NOT a fixed huge sleep). Guard: SendInput does NOT reset key state - check GetAsyncKeyState for stuck Ctrl/Win from the still-held PTT chord and synthesize key-ups first. 4. Rung C - SendInput KEYEVENTF_UNICODE (0x0004) typing (wVk=0, wScan=UTF-16 unit; surrogate-aware, 2 INPUTs/BMP unit, batched into ONE SendInput; tag dwExtraInfo). Fallback when paste is blocked; avoid into legacy conhost pre-v1.16 (VK_PACKET mangles emoji) and active IME. 5. Rung D - decline + leave-on-clipboard + toast 'Dictation copied, press Ctrl+V' (secure/elevated/exclusive-fullscreen games). Per-app override table keyed by exe/AppUserModelID. Chunked paste (250/500/750/1000, default 250, split on whitespace) for terminal AI-CLI agents. || OVERLAY WINDOW FLAGS: WebviewWindowBuilder (Handy's exact non-macOS recipe): title 'Recording', resizable(false), inner_size ~256x46 tight to the pill, shadow(false) [THE critical flag - shadow-on is the #8308 'opaque box' bug], maximizable(false), minimizable(false), closable(false), accept_first_mouse(true) [macOS no-op, harmless], decorations(false), always_on_top(true), skip_taskbar(true), transparent(true) [MUST be set at creation, not toggled], focusable(false), focused(false), visible(false). tao maps focusable(false) -> WS_EX_NOACTIVATE (0x08000000, non-activating: target app keeps keyboard focus so injected text lands, buttons still get WM_LBUTTONDOWN), always_on_top -> WS_EX_TOPMOST (0x00000008), ignore_cursor_events -> WS_EX_TRANSPARENT (0x00000020) | WS_EX_LAYERED (0x00080000) [whole-window only; no per-region hit-test, Tauri #13070 - use Handy's tight-window design, NOT click-through polling]. Re-assert Z-order after show via force_overlay_topmost() SetWindowPos(HWND_TOPMOST, SWP_NOMOVE|SWP_NOSIZE|SWP_NOACTIVATE|SWP_SHOWWINDOW). Position bottom-center on the monitor WORK AREA (MONITORINFO.rcWork / SPI_GETWORKAREA), not the full rect, so it clears the taskbar; detect auto-hide via SHAppBarMessage ABM_GETSTATE ABS_AUTOHIDE (0x1) and add a ~48px bottom inset. DPI: rely on Tauri's default PerMonitorV2 manifest (DPI_AWARENESS_CONTEXT_PER_MONITOR_AWARE_V2); handle WM_DPICHANGED/WM_DISPLAYCHANGE and fractional scaleFactor. Transparency mechanism: tao calls DwmEnableBlurBehindWindow with an empty region; WebView2 DefaultBackgroundColor=alpha 0 shows desktop through; semi-transparent pill via CSS rgba() over that (semi-transparent DEFAULT bg is unsupported - alpha must be 0 or 255). Set WEBVIEW2_DEFAULT_BACKGROUND_COLOR=00FFFFFF at startup if a first-frame white flash appears. Known multi-monitor gotcha (Handy #811): overlay can be invisible on secondary monitors - test. || TRAY: Tauri TrayIconBuilder (Win32 Shell_NotifyIcon underneath). Win11 hides new tray icons in the '^' overflow with no programmatic promotion - onboarding must tell users to drag it out, and core function must never depend on visibility (hotkey works regardless). || AUTOSTART: tauri-plugin-autostart -> HKCU Run key HKEY_CURRENT_USER\\SOFTWARE\\Microsoft\\Windows\\CurrentVersion\\Run (per-user, no admin; appears in Task Manager Startup). Default OFF, Settings toggle drives enable()/disable(). Enforced at the audio stack: when off, WASAPI/IAudioClient returns silence/E_ACCESSDENIED-class failure. Registry: HKCU\\Software\\Microsoft\\Windows\\CurrentVersion\\CapabilityAccessManager\\ConsentStore\\microphone\\NonPackaged\\Value = Allow/Deny. There is NO requestAccess API - detect blocked state (open capture -> silence/zeros, or read the registry Value==Deny) and deep-link to ms-settings:privacy-microphone via the opener plugin. Onboarding copy must branch per-OS. || INSTALLER + SIGNING + SMARTSCREEN/DEFENDER PLAN: NSIS -setup.exe with installMode: currentUser (=> %LOCALAPPDATA%, no UAC, matches Wispr's per-user model); ship NSIS-only for the auto-updater path (mixing MSI creates a second uninstaller and NSIS->MSI upgrade is not clean). Silent flag /S. Signing = Azure Trusted Signing (trusted-signing-cli), the same cheap cloud path Wispr uses (~72h short-lived leaf certs, chain Microsoft ID Verified CS AOC CA 04 -> ... 2020 root; RFC3161 timestamp for post-expiry trust; Authenticode SHA-256). SmartScreen: a fresh signing identity has zero reputation -> 'Windows protected your PC' until download volume accrues; mitigate by signing every release consistently, submitting the binary, and accumulating reputation. Defender: the sidecar's WH_KEYBOARD_LL hook + SendInput + clipboard is heuristically keylogger/RAT-shaped and is the #1 false-positive trigger - sign BOTH the app exe and the sidecar exe, do not pack/obfuscate, and submit samples to Microsoft (MSRC/WDSI) for analyst allowlisting before launch. Optional uiAccess='true' build (manifest, signed, installed under %ProgramFiles%) as an opt-in installer mode to drive elevated windows - but that requires per-machine install, breaking the per-user default. || AUTO-UPDATE: tauri-plugin-updater 2.10.0, one mechanism both OSes. Static latest.json (version, notes, pub_date, platforms.'windows-x86_64'.{url,signature}); ed25519/minisign signature (pubkey in tauri.conf.json plugins.updater.pubkey, TAURI_SIGNING_PRIVATE_KEY env; verification cannot be disabled). windows.installMode: passive (default). Flow check()->download()->install()->relaunch() (Windows app auto-exits on install). || ASR + LLM BACKEND TIERS BY HARDWARE (auto-detected via DXGI EnumAdapters1 VendorId 0x10DE NVIDIA/0x1002 AMD/0x8086 Intel + DedicatedVideoMemory, Vulkan deviceType, GlobalMemoryStatusEx RAM, __cpuidex leaf 7 AVX2 bit5/AVX512 bit16). ## Risk register

R1 HOOK TIMEOUTS / SILENT REMOVAL (Windows, HIGH). WH_KEYBOARD_LL callback exceeding LowLevelHooksTimeout (HKEY_CURRENT_USER\\Control Panel\\Desktop, default 5000ms, clamped to max 1000ms since Win10 1709) is SILENTLY removed with no notification (Microsoft: undetectable by design); handy-keys only re-hooks on suspend/session events. Co-resident ONNX+llama.cpp can starve the hook thread past 1000ms. MITIGATION: run the hook in the SEPARATE Rust sidecar process (cannot be starved by inference), keep the callback to a channel-send only (single-digit-us), raise the sidecar/hook-thread priority above inference, cap llama.cpp/ort thread pools to leave >=1 core, and shell-heartbeat the sidecar to respawn a hung helper. macOS CGEvent tap-timeout is recoverable (kCGEventTapDisabledByTimeout->CGEventTapEnable + 100ms health poll + full reinstall). || R2 DEFENDER / SMARTSCREEN FLAGS (Windows, HIGH). A global keyboard hook + SendInput + clipboard automation is keylogger/RAT-shaped; a fresh Azure Trusted Signing identity has zero SmartScreen reputation. MITIGATION: Authenticode-sign BOTH the app exe and the sidecar exe; no packing/obfuscation; submit samples to Microsoft (MSRC/WDSI) for analyst allowlisting pre-launch; accumulate download reputation; keep the sidecar's native surface minimal and auditable. || R3 iGPU / CPU LLM LATENCY (Windows, MEDIUM-HIGH). 4B Q4 on Intel Iris Xe (~10-14 tg, ~5-7s) and CPU-only 8-core (~16-18 tg, ~4-4.5s) BUSTS the perceived budget; 8GB machines page. MITIGATION: hardware-tiered model selection (drop to Qwen3-1.7B/Llama-3.2-1B on weak tiers, all-Qwen ladder), streaming to hide TTFT (~0.35-0.7s perceived), tiered download so weak machines never fetch the 4B, and Claude as the zero-download fallback; gate local LLM on AVX2 and >=16GB for the 4B. || R4 WEBVIEW RENDERING PARITY (both, MEDIUM). Tauri transparency 'opaque box' bug (#8308) is caused by shadow-on-by-default; black-EDGE artifacts (#14823/#14764) are resize/decoration/focus-bound. MITIGATION: Handy's exact recipe - transparent at CREATION, shadow(false), decorations(false), unparented, non-resizable tight 256x46 window; animate hover-expand via CSS inside a fixed window, not runtime resize; set WEBVIEW2_DEFAULT_BACKGROUND_COLOR=00FFFFFF if white-flash. FLIP-TO-ELECTRON gate: only if a static shadow-off pill goes black at rest on real Intel/NVIDIA/AMD Win11 (unverified, contradicted by the issue corpus). Also test Handy #811 (overlay invisible on secondary monitors). || R5 EXCLUSIVE-FULLSCREEN OVERLAY (both, MEDIUM, mostly unfixable). True DXGI exclusive-fullscreen (and macOS equivalent) bypasses the compositor; no WS_EX_TOPMOST / NSPanel level beats it (identical for Electron - a wash). MITIGATION: works over borderless/flip-model fullscreen (most modern games + Windows fullscreen-optimizations); re-assert HWND_TOPMOST on state transitions; document 'pill hidden during exclusive-fullscreen games; use borderless/windowed'; leave-on-clipboard fallback. || R6 ELEVATED WINDOWS / UIPI (Windows, MEDIUM). A Medium-IL app cannot hook or SendInject into a High-IL (Run-as-admin) foreground window; injection fails SILENTLY (neither GetLastError nor return value signals UIPI). MITIGATION: detect target integrity level (OpenProcess+GetTokenInformation, HIGH 0x3000 vs MEDIUM 0x2000) and decline+toast 'copied, press Ctrl+V; run WhimprFlow as admin to type here'; offer an optional uiAccess='true' signed %ProgramFiles% installer mode for parity (cost: per-machine install). macOS analog = secure-input (IsSecureEventInputEnabled) naming toast. || R7 ASR BUDGET ON MID-TIER MAC (macOS, MEDIUM). ort-CPU Parakeet misses ~300-500ms on base M-series/M1. MITIGATION: FluidAudio ANE primary behind the trait (the whole FORK-1 decision); ort-CPU only as portable fallback. || R8 CoreML EP DEAD-END (macOS, LOW but decision-load-bearing). Enabling ort-coreml gives NO ANE for Parakeet-TDT (LSTM + int8 QDQ unsupported) and can regress the decoder loop. MITIGATION: never rely on the CoreML EP for Parakeet; use native FluidAudio for ANE. || R9 tao/Tauri OVERLAY PATCHES (both, LOW). Handy pins tao to an unreleased rev via [patch]; Tauri's window layer occasionally needs source patches. MITIGATION: pin tao to Handy's known-good rev, budget for a [patch] entry, track Tauri overlay issues. || R10 DUAL-BINARY RELEASE DRIFT (both, LOW-MEDIUM). Sidecar + shell must version-match and keep IPC-schema compat across auto-update. MITIGATION: codegen the stdio JSON schema (quicktype-style), atomic bundle update via tauri-plugin-updater, IPC version handshake on spawn. || R11 SIDECAR IPC LATENCY (both, LOW). Every key transition + paste crosses a process boundary. MITIGATION: length-prefixed JSON over stdio is sub-ms to low-ms; keep paste ordering/clipboard timing in the sidecar (not round-tripped); it is non-critical for hold-to-talk. || R12 WIN-ON-ARM (Windows, LOW). Per-arch ONNX/whisper backend differs (Handy #1682: aarch64 GPU transcription fails). MITIGATION: ship CPU aarch64 ggml + arch-specific ort selection; treat Win-ARM GPU as best-effort.
